Storage and Retrieval - Transaction Processing or Analytics?

在古老的从前，database的write一般是和commercial transaction有关。到现在，transaction这个名号保留了下来，用于form一个logical unit，里面包含一系列read、write操作。

有一些应用很像处理business transaction，根据key来取一些record，根据和用户的交互来插入或更新一些record，这种模式被称为online transaction processing（OLTP）。

有一些database专注于数据分析，通常需要scan大量的数据，每条record读取少量的column，计算一些aggregate statistic（比如count、sum或者average），这种模式被称为online analytic processing（OLAP）。

它们之间有以下差别：

Property OLTP OLAP
Main read pattern Small number of records per query, fetched by key Aggregate over large number of records
Main write pattern Random-access, low-latency writes from user input Bulk import (ETL) or event stream
Primarily used by End user/customer, via web application Internal analyst, for decision support
What data represents Latest state of data (current point in time) History of events that happened over time
Dataset size Gigabytes to terabytes Terabytes to petabytes

以前处理OLTP和OLAP都是用同样的database，后来OLAP采用另外（separate）的database，称为data warehouse。

Data Warehousing

Data warehouse是不会影响到应用的OLTP的分离出去的database。它需要从所有可用的OLTP里读取数据的read-only copy。过程是：

  • 数据先从OLTP database中extract。
  • transform到analysis-friendly schema。
  • clean up并load到data warehouse中。

整个过程被称为Extract-Transform-Load（ETL）。

Stars and Snowflakes: Schemas for Analytics

Analysis过程的data model不如transaction processing的那么丰富，大多数data warehouse都采用star schema（也叫作dimensional modeling）。

最中间的schema叫做fact table，每一行代表一件在过去特定时间里发生过的事情。fact table中有一些column是attribute，有一些则是foreign key reference。fact table中的每一row都代表一个event，而dimension则代表event的who, what, where, when, how, 和why。

这种模式还有一种变体，称为snowflake schema，即每一dimension还会进行further broke down到subdimension，像雪花一样。Snowflake schema比star schema更normalized，但star schema对于analyst来说更加简单。

Fact table在实际中通常会有超过100个column，因为每条fact需要记录非常多的细节。

而为了加速这个过程，有一种思想是column-oriented storage。不要按一行行地存储，而是每个column都保存到一起，这样使得对于特定column的query相当有效。

Writing to Column-Oriented Storage

Column-oriented storage, compression, and sorting all help to make those read queries faster. However, they have the downside of making writes more difficult.

Aggregation: Data Cubes and Materialized Views

Data warehouse的query经常包含aggregate function，比如SQL中的COUNT、SUM、AVG，MIN或者MAX。可以尝试cache一些aggregate值，来加速query。这种cache被称作materialized view。当底层数据改变，materialized view也跟着更新。

Materialized的一种特例叫做data cube或者叫OLAP cube。它能将不同dimension的数据进行aggregate。

这个例子是2个dimension，aggregating data by summing。对于2个dimension的情况，人类尚能想象出来，3个dimension以上就比较困难了，但是和2个dimension的原理是一样的，都能cache一些aggregate值，从而加速query过程，而无需scan上百万行。
